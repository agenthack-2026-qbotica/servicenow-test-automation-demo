## Issue Summary

**Issue #55 ("Test23"):** ServiceNow must refuse to resolve an incident when the **Resolution Notes** field is left empty — the state change to `Resolved` should be blocked, a validation message should be shown, and the incident must remain in its prior state. There was no automated test covering this requirement.

**About the attached "user request":** The root-cause text supplied with this task references selector/timing failures in `TestCases\01_TC_CreateIncident.xaml` and a file called `TestCases\04_TC_ResolveHighPriorityWAssignGroup.xaml`. Neither claim matches the current repository: `01_TC_CreateIncident.xaml` has no failing run evidence attached, and no file named `04_TC_ResolveHighPriorityWAssignGroup.xaml` exists (the real file is `04_TC_HighPriorityWAssignGroup.xaml`, and it already resolves incidents correctly on the happy path). This looks like a generic/mismatched template rather than real failure evidence for this issue, so this fix is driven by the actual **ISSUE DESCRIPTION** (Test23 acceptance criteria) instead.

**Root cause:** Walking the suite with the `uipath-review` / `uipath-troubleshoot` conventions turned up two concrete problems:

1. **No test coverage existed for the acceptance criteria at all.** `TestCases/` only went up to `08_TC_OpenIncident.xaml`; nothing exercised "resolve an incident with empty Resolution Notes." The Object Repository (`.objects/_70W/XuB5/1IAJ/...`, app `_BU5etMzk0KtDV4Hwc6ztw`) already contains every UI element needed for this flow — `Button_ResolutionInformation`, `Button_ResolutionCode`, the `Resolution notes` textarea, `Button_Resolve`, and `Text_Resolved` — and they're used by `Workflows/04_HighPriorityWAssignGroup.xaml` / `05_HighPriorityWoAssignGroup.xaml`, but only on the happy path where notes **are** filled in before clicking Resolve. The negative-path scenario the issue describes was simply never wired up.

2. **A stale, mislabeled test case that pointed at the wrong thing.** `TestCases/06_TC_UpdateCaller.xaml` — which actually updates the incident's Caller field — still carried `x:Class="_06_CloseIncidentWoResolutionNotes"`, `DisplayName="_06_CloseIncidentWoResolutionNotes"`, and `IdRef="_06_CloseIncidentWoResolutionNotes_1"`, even though its underlying workflow (`Workflows/06_UpdateCaller.xaml`) is correctly named `_06_UpdateCaller`. This confirms `06_TC_UpdateCaller.xaml` began life as the stub for this exact Resolution-Notes scenario (matching the stub name recorded in `CLAUDE.md`, `06_TC_CloseIncidentWoResolutionNotes.xaml`) and was later repurposed into the Update-Caller test without cleaning up its internal identifiers. A test case whose class/display name advertises "resolution notes" behavior while its body drives an unrelated flow is exactly the kind of mismatch that produces confusing, uninterpretable results in Test Manager/CI.

## UiPath Analysis

- Every test case in this project follows a strict Given/When/Then structure (`SNOW_Login.xaml` → action workflow → assertion(s) → `SNOW_Logout.xaml`), and verification is done with `UiPath.Testing.Activities.VerifyControlAttribute` wrapping an `NGetText` against a `Home_Dashboard` `NApplicationCard` — exactly as in `TestCases/04_TC_HighPriorityWAssignGroup.xaml` and `05_TC_HighPriorityWoAssignGroup.xaml` (which verify the *positive* resolve case: `Text_Resolved != ""`). The new test case mirrors this pattern with the assertion polarity inverted (`Text_Resolved == ""`), since resolution should be blocked here.
- The Object Repository app (`_BU5etMzk0KtDV4Hwc6ztw`) already has every element needed for the "attempt to resolve" half of this scenario captured under the `Home_Dashboard` screen (`1IAJgRc-uECKmXeotj3r1Q`): `Button_ResolutionInformation`, `Button_ResolutionCode`, `Resolution notes` (input), `Button_Resolve`, and `Text_Resolved`. These are reused by `Reference` rather than re-captured, consistent with Studio's Object Repository sync behavior and keeping selector maintenance centralized.
- The one selector that could **not** be captured from a live PDI in this environment is ServiceNow's client-side mandatory-field validation message shown when Resolution Notes is empty at Resolve time. A best-effort fuzzy selector (`innertext` matching `*Resolution notes*required*`) was used, based on ServiceNow's documented out-of-box "resolution notes required" business rule wording. That assertion runs with `ContinueOnFailure="True"` and screenshot-on-failure so a real run fails loudly (with evidence) rather than silently passing if the exact wording/selector needs adjustment.
- The `TestCases/06_TC_UpdateCaller.xaml` naming mismatch is a second, independent defect: if any tooling resolves a test run by the workflow's internal class/display name rather than its file path, it would report results under "resolution notes" while actually running "update caller" logic — producing misleading history with no relation to the story being investigated. Fixing the identifiers removes that ambiguity without changing runtime behavior.
